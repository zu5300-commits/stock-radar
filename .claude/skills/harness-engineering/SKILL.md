# Harness Engineering(馬具工程)

決定一個 AI 能不能**可靠地**把事情做好,通常不是模型本身,而是包在模型外面的那一整套系統——**馬具(harness)**。模型是引擎,馬具才是整台車。馬具不會讓模型「更聰明」,它把模型既有的能力**可靠地導向目標**。

→ 推論:面對 agent 表現不佳,**先別等下一代模型,也先別急著換模型**。優先檢查並改進馬具,現有模型往往就能變強。

## 何時套用這個 skill

出現以下任一情況時,套用本方法論:

- 要從零打造或改進一個 agent / 自動化流程 / MCP 工具集 / RAG / agentic pipeline。
- agent 行為不穩:同問題兩次答不同、自信地給錯、做到一半忘了前文、產出有 bug 但自己沒發現、重複修同一個檔案鬼打牆。
- 有人把問題歸因成「模型不夠強」——先用馬具六要素檢查是不是馬在裸跑。

## 馬具六要素(設計與診斷的檢查清單)

逐項檢查;**缺一項就是馬在裸跑**。

1. **迴圈 (Loop)** — 是否「觀察 → 決定 → 動手 → 驗證 → 修正」一圈圈轉到完成,而不是回一句話就停?
   - 診斷訊號:做完就走、不會根據結果自我修正。
2. **工具 (Tools)** — 是否不只會「講」還會「做」(讀檔、跑程式、呼叫 API、查網路)?工具描述是否清楚、可被正確選用?
3. **上下文 (Context)** — 是否控制了「看到什麼/不看什麼」?有沒有被無關資訊淹沒、或反之缺了關鍵脈絡(例如沒先把專案/環境掃過餵進去)?
4. **記憶 (Memory)** — 跨對話、跨步驟是否記得做過什麼?該持久化的有沒有落地?
5. **驗證 (Verification)** — 做完是否自己跑測試 / 自我審查,而不是嘴上說「做完了」就交差?
6. **限制 (Constraints)** — 邊界清不清楚:哪些檔案/操作不准碰、花費與步數上限、失敗時的行為?

> 關鍵:這六項**沒有一項是「模型」的改進,全是「系統工程」的改進**。

## 核心原則

- **少即是多。** 工具/選項太多會讓 agent 迷路。實測:把工具從一大堆砍到只剩必要的幾個,準確率反升、速度更快、成本更低。**「限制」本身是馬具最重要的一環。**
- **限制 > 自由。** 給清楚的邊界(禁區、上限、失敗策略)比給更多能力更能提升可靠度。
- **環境優於意志力。** 要改變行為,改「環境/限制/迴圈」比「升級核心能力」更快見效。
- **前提:引擎要夠力。** 馬具不能救一個連基本邏輯都跑不動的爛模型;模型夠強時,馬具的價值才浮現。
- **做馬具不性感但有效。** 瑣碎、龜毛、要試錯,但真正拉開差距的就是這種苦工。

## 建議流程

設計或審查一個 agent 時:

1. **先框定問題**:把「該換哪個模型?」改寫成「我的馬具哪一項沒做好?」
2. **逐項過六要素**:用上面的清單標出缺失或薄弱的環節。
3. **先做減法**:砍掉非必要的工具與上下文,收緊限制,再考慮加東西。
4. **補上迴圈與驗證**:確保有「動手 → 驗證 → 修正」的閉環,而不是單次輸出。
5. **餵對上下文**:開工前把必要的專案/環境資訊掃過餵進去;擋掉無關雜訊。
6. **量測再迭代**:用準確率 / 速度 / 成本三個面向驗證每次調整,而非憑感覺。

## 一句話結論

別再問「哪個模型最強」,改問:**「我的馬具,做好了沒?」**——不管是 AI agent,還是任何要可靠交付的系統。
